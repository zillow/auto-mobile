# Extract shared test-plan-validation library

## Problem

Test plan YAML validation logic is currently duplicated across multiple locations:

1. **junit-runner** (`android/junit-runner/src/main/kotlin/.../PlanSchemaValidator.kt`) - 252 lines
2. **IDE plugin** (`android/ide-plugin/src/main/kotlin/.../TestPlanValidator.kt`) - 300+ lines
3. **Tool name list** - duplicated in both locations (59 tools)

This duplication leads to:
- ❌ Maintenance burden: changes must be made in 2+ places
- ❌ Inconsistency risk: different bugs could emerge in different validators
- ❌ Tool registry drift: new tools must be added to multiple files
- ❌ Missing features: junit-runner lacks tool name validation that IDE plugin has

## Proposed Solution

Create a lightweight **internal** validation library that both junit-runner and IDE plugin depend on.

```
android/test-plan-validation/
├── build.gradle.kts
├── src/
│   ├── main/
│   │   ├── kotlin/com/automobile/validation/
│   │   │   ├── TestPlanValidator.kt        # Core validation logic
│   │   │   ├── ToolRegistry.kt             # Valid tool names (59 tools)
│   │   │   └── Models.kt                   # ValidationResult, ValidationError
│   │   └── resources/
│   │       └── schemas/
│   │           └── test-plan.schema.json   # Schema (copy or symlink)
│   └── test/
│       └── kotlin/com/automobile/validation/
│           └── TestPlanValidatorTest.kt    # Shared validation tests
```

### Module Structure

**build.gradle.kts:**
```kotlin
plugins {
  kotlin("jvm")
  kotlin("plugin.serialization")
  `java-library`
}

dependencies {
  // YAML processing and schema validation
  implementation("org.yaml:snakeyaml:2.3")
  implementation("com.networknt:json-schema-validator:1.5.3")
  implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.7.3")

  testImplementation("org.jetbrains.kotlin:kotlin-test")
  testImplementation("junit:junit:4.13.2")
}

// NOT published to Maven Central - internal use only
```

**ToolRegistry.kt:**
```kotlin
package com.automobile.validation

/**
 * Registry of valid AutoMobile tool names.
 * Single source of truth for tool validation across junit-runner and IDE plugin.
 */
object ToolRegistry {
    val VALID_TOOLS: Set<String> = setOf(
        // App management
        "launchApp", "terminateApp", "listApps", "installApp",
        // UI interactions
        "tapOn", "swipeOn", "pinchOn", "dragAndDrop",
        // ... all 59 tools
    )

    /**
     * Check if a tool name is valid
     */
    fun isValidTool(toolName: String): Boolean = toolName in VALID_TOOLS
}
```

**TestPlanValidator.kt:**
```kotlin
package com.automobile.validation

/**
 * Validates AutoMobile test plan YAML files against JSON schema.
 * Shared validation logic used by junit-runner and IDE plugin.
 */
object TestPlanValidator {
    fun validateYaml(yamlContent: String): ValidationResult {
        // Unified validation logic:
        // 1. Parse YAML
        // 2. Validate against schema
        // 3. Validate tool names
        // 4. Check for deprecated fields
        // 5. Return structured errors with line/column info
    }
}
```

**Models.kt:**
```kotlin
package com.automobile.validation

enum class ValidationSeverity { ERROR, WARNING }

data class ValidationResult(
    val valid: Boolean,
    val errors: List<ValidationError> = emptyList()
)

data class ValidationError(
    val field: String,
    val message: String,
    val severity: ValidationSeverity = ValidationSeverity.ERROR,
    val line: Int? = null,
    val column: Int? = null
)
```

## Benefits

### 1. Single Source of Truth
- ✅ One place to maintain validation logic
- ✅ One place to update tool registry when adding new tools
- ✅ Consistent behavior across IDE and JUnit tests

### 2. Feature Parity
- ✅ junit-runner gets tool name validation automatically
- ✅ junit-runner gets deprecated field warnings automatically
- ✅ All consumers benefit from validation improvements

### 3. Reduced Maintenance
- ✅ ~500 lines of duplicated code removed
- ✅ Schema bundled once, not duplicated
- ✅ Tests consolidated in one place

### 4. Future Extensibility
- ✅ Easy to add new validation rules (e.g., selector validation)
- ✅ Could be used by other tools (CLI, web validator, etc.)
- ✅ Foundation for more sophisticated validation (cross-step checks, etc.)

## Migration Plan

### Phase 1: Create Module
1. Create `android/test-plan-validation` module
2. Extract core validation logic from IDE plugin validator
3. Include tool registry with all 59 tools
4. Add comprehensive tests (combine tests from both locations)
5. Bundle schema file

### Phase 2: Migrate junit-runner
1. Add dependency: `implementation(project(":test-plan-validation"))`
2. Replace `PlanSchemaValidator` with wrapper around shared validator
3. Update tests to verify same behavior
4. Remove duplicated code (~250 lines)

### Phase 3: Migrate IDE Plugin
1. Add dependency to `android/ide-plugin/build.gradle.kts`
2. Replace `TestPlanValidator` with wrapper that adds IDE-specific features:
   - IntelliJ-specific error positioning
   - Quick fixes (handled by IDE plugin, not shared validator)
   - Real-time validation hooks
3. Remove duplicated validation logic (~200 lines)

### Phase 4: Documentation
1. Document shared validation architecture
2. Update contributor guide: "Adding a new tool? Update ToolRegistry.kt"
3. Add ADR (Architecture Decision Record) for validation library

## Non-Goals

- ❌ **NOT** published to Maven Central (internal use only)
- ❌ **NOT** used by TypeScript MCP server (different runtime, but could inform schema)
- ❌ **NOT** a general-purpose YAML validator (AutoMobile-specific only)

## Notes

The validation library will be **internal** (not published) because:
1. It's tightly coupled to AutoMobile's schema and tools
2. No external consumers need this functionality
3. Avoids Maven Central publishing overhead
4. Keeps versioning simple (internal use only)

However, the module structure makes it easy to publish later if needed.
